## TL;DR
This project is a sophisticated AI Agent Framework designed to orchestrate LLM interactions with a heavy emphasis on **tool-use**, **state persistence**, and **content optimization**. It implements a custom "Artifact" system for structured output and a "Caveman" compression utility to reduce token overhead. The architecture is primarily an asynchronous FastAPI backend that bridges a request/response API with a long-running agent loop capable of interacting with the local filesystem and external web services.

## Design Overview
*   **Orchestrated Agent Loop:** The system separates the API layer (`api/`) from the agent's execution logic (`agent/loop.py` and `agent/session_runner.py`), allowing the agent to run as a stateful process independently of the HTTP connection.
*   **Provider Agnostic Strategy:** By using a "Provider" pattern (`api/providers.py`, `providers/ollama_provider.py`), the system can swap between different LLM backends (Ollama, Anthropic, OpenAI) while maintaining a consistent internal interface.
*   **Resource Guarding & Security:** A consistent pattern of "path guarding" and "SSRF prevention" is applied across tools (especially in `tools/krawlx.py` and `api/skills_admin.py`) to prevent the agent from accessing unauthorized system files or internal network targets.
*   **Token Economy:** The "Caveman" module implements a custom markdown compression algorithm to maximize the information density passed to the LLM, reducing costs and latency.

## Concepts & Patterns

**Strategy Pattern (Provider Resolution)**
- **What**: Defining a family of algorithms (LLM providers) and making them interchangeable.
- **Why used here**: To support Ollama, Anthropic, and OpenAI without rewriting the agent loop logic.
- **Trade-offs**: Adds a layer of abstraction; requires a "lowest common denominator" approach to tool-calling formats.
- **When to use**: When your application depends on third-party APIs that offer similar functionality but different interfaces.

**Asynchronous Event Loop (FastAPI + Asyncio)**
- **What**: Non-blocking execution of tasks.
- **Why used here**: The system must handle long-running LLM streams, file I/O, and web crawling without freezing the API server.
- **Trade-offs**: Increases complexity (async/await everywhere); requires careful handling of thread-safe shared state (e.g., `MemoryManager`).
- **When to use**: High-I/O applications or real-time streaming services.

**SSRF (Server-Side Request Forgery) Protection**
- **What**: Validating that a requested URL does not point to internal IP addresses (127.0.0.1, etc.).
- **Why used here**: The `KrawlXTool` allows the AI to fetch URLs; without this, the AI could be tricked into attacking the host server.
- **Trade-offs**: Slightly more latency during URL validation.
- **When to use**: Always when an application fetches remote content based on user or AI input.
