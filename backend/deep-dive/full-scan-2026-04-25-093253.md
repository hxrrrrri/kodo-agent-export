## TL;DR
AntiVibe is a sophisticated AI agent orchestration framework designed to bridge the gap between high-level LLM reasoning and low-level system execution. It transforms a standard chat interface into a "command center" capable of executing bash commands, managing its own memory, interacting with external databases, and employing a "Code Review Graph" for structural codebase analysis. It exists to provide a secure, observable, and extensible runtime where AI agents can perform complex engineering tasks with a human-in-the-loop safety model.

## Design Overview
*   **Agentic Loop Architecture**: Instead of simple request-response, it uses a `SessionRunner` and `AgentLoop` (agent/loop.py) that enables iterative tool use, where the agent can observe the output of a tool and decide the next step.
*   **Secure Sandbox Pattern**: Implementation of a `path_guard` (tools/path_guard.py) and `PermissionHub` (api/permission_hub.py) to prevent "Zip Slip" attacks, directory traversal, and unauthorized system access.
*   **Provider Agnostic Layer**: A strategy-based provider system (providers/smart_router.py) that abstracts different LLM backends (Ollama, Gemini, Groq, DeepSeek) and allows for dynamic routing based on latency and quality.
*   **Multi-Modal Tooling**: Integration of diverse capabilities ranging from `MCP (Model Context Protocol)` servers to a specialized `Caveman` compressor for reducing token overhead in large files.

## Code Walkthrough

### `api/` — The Interface Layer
This layer handles the HTTP boundary using **FastAPI**.
*   `chat.py` (L120-L164): The heart of the API. It manages session state, extracts project directories, and handles the streaming response from the agent loop.
*   `security.py` (L12-L96): Implements a `SlidingWindowRateLimiter` to prevent API abuse, ensuring stability under load.
*   `providers.py` & `profiles.py`: Manage the "intelligence" settings, allowing users to switch between different LLM profiles without restarting the server.

### `agent/` — The Reasoning Engine
This is where the "brain" of the system lives.
*   `loop.py` (L66-L240): Manages the `RuntimeConfig`. It converts LLM tool calls into actual Python function executions and handles the conversation history truncation to stay within context windows.
*   `session_runner.py` (L23-L172): Orchestrates the lifecycle of a session, including automatic title generation and persistence of session results.
*   `coordinator.py`: Handles the spawning and stopping of sub-agents, enabling a hierarchical agent structure.

### `tools/` — The Capability Set
Every file here is a specialized "skill" the agent can invoke.
*   `bash.py` & `powershell.py`: Provide raw system access with cross-platform translation (e.g., translating `ls` to PowerShell commands on Windows).
*   `krawlx.py`: A high-performance web crawler with SSRF protection (preventing the agent from attacking local network services).
*   `repl.py`: Implements persistent Python and Node.js sessions, allowing the agent to maintain state across multiple code executions.
*   `mcp_tool_call.py`: Implements the Model Context Protocol, allowing the agent to connect to external tool registries.

### `artifacts/` & `memory/` — The State Layer
*   `store.py` (L29-L183): Manages "Artifacts" (files or snippets the agent creates) with versioning and LRU eviction to prevent disk saturation.
*   `manager.py` (L36-L397): A persistent key-value store that allows the agent to "remember" facts across different chat sessions.

## Concepts & Patterns

**Strategy Pattern (Provider Routing)**
- **What**: Defining a family of algorithms (LLM Providers) and making them interchangeable.
- **Why used here**: Enables the `SmartRouter` to switch from a slow, high-quality model (e.g., Gemini) to a fast, local model (Ollama) based on the task.
- **Trade-offs**: Increases complexity in the configuration layer but eliminates vendor lock-in.
- **When to use**: When the system must support multiple external APIs with similar interfaces but different performance characteristics.

**Middleware/Guard Pattern (Path Guard)**
- **What**: A verification layer that intercepts all file system requests to ensure they stay within an allowed "root" directory.
- **Why used here**: To prevent the LLM from accidentally (or maliciously) reading `/etc/passwd` or writing to system binaries.
- **Trade-offs**: Slight performance hit on every I/O operation.
- **When to use**: Any application where an AI agent has write/read access to a local filesystem.

**Asynchronous Stream Processing**
- **What**: Using `asyncio` and `StreamingResponse` to push tokens to the client as they are generated.
- **Why used here**: LLM responses are slow; streaming ensures the user sees progress in real-time rather than waiting 30 seconds for a complete block.
- **Trade-offs**: More complex error handling (you can't change the HTTP status code after the first byte is sent).
- **When to use**: Real-time AI applications and long-running task execution.
