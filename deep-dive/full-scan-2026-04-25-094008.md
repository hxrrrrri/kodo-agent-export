## TL;DR
This project is a sophisticated, multi-lingual AI orchestration platform designed to bridge the gap between high-level LLM reasoning and low-level system execution. It combines a **FastAPI/Python** backend for agentic tool-use and memory management, a **React/TypeScript** frontend for an interactive "Artifact-based" UI, and a high-performance **Rust** core for native document processing and HTML parsing. Essentially, it transforms a chat interface into an integrated development environment (IDE) where AI can crawl the web, write code, execute shells, and render complex visual artifacts.

## Design Overview
*   **Polyglot Orchestration**: Uses Python for the "brain" (agent loops, tool orchestration), Rust for "muscle" (high-speed parsing, PDF/Docx processing), and TypeScript for the "face" (dynamic UI rendering).
*   **Artifact-Centric UX**: Instead of linear chat, the system uses an "Artifact" pattern where code or data generated by the AI is extracted into a persistent side-panel with specialized runtimes (React, HTML, Mermaid, etc.).
*   **Agentic Tool-Loop**: Employs a sophisticated `AgentLoop` that can autonomously decide to use tools (Bash, Git, WebSearch, MCP) and manage its own state through a `MemoryManager`.
*   **Distributed Web Extraction**: Integrates deeply with `Firecrawl` for high-fidelity web scraping, transforming raw HTML into LLM-ready markdown via a multi-stage pipeline.

## Code Walkthrough

### Backend Logic (`backend/`)
- **`agent/loop.py` & `coordinator.py`**: The core engine. The `AgentLoop` manages the conversation history, handles tool-calling logic, and interfaces with various LLM providers. The `Coordinator` manages the lifecycle of multiple agents.
- **`tools/`**: A modular library of system capabilities. `bash.py`, `git_tool.py`, and `file_edit.py` provide the AI with "hands" to manipulate the local filesystem and repository.
- **`api/`**: A FastAPI-based gateway. `chat.py` handles the streaming LLM responses, while `antivibe.py` and `artifacts.py` manage the state of the visual components on the frontend.
- **`providers/`**: An abstraction layer for LLMs. `smart_router.py` allows the system to switch between models (Claude, OpenAI, Ollama) based on latency, cost, or quality.

### Frontend Experience (`frontend/src/`)
- **`components/ChatWindow.tsx` & `MessageBubble.tsx`**: The primary interface. It doesn't just render text; it identifies "Artifact" blocks in the LLM stream and triggers the side-panel.
- **`components/artifacts/`**: A suite of runtimes. `ReactRuntime.tsx` and `HtmlRuntime.tsx` use iframes and sandboxing to execute AI-generated code safely in the browser.
- **`store/chatStore.ts`**: A Zustand-based state management system that tracks conversation history and the active "Artifact" versions.
- **`hooks/useChat.ts`**: The bridge between the UI and the FastAPI backend, handling Server-Sent Events (SSE) for real-time typing effects.

### Native Core (`.tmp/firecrawl-src/apps/api/native/`)
- **`src/html.rs` & `src/document/`**: Written in Rust for performance. These files handle the heavy lifting of parsing complex binary formats (DOCX, XLSX, PDF) and stripping HTML noise, providing the "clean" data the LLM needs to avoid token waste.

## Concepts & Patterns

**The Artifact Pattern**
- **What**: Separating the "conversation" (chat) from the "work product" (code/UI).
- **Why used here**: Prevents the chat history from becoming cluttered with 1,000-line code blocks, allowing the user to iterate on a versioned document in a dedicated view.
- **Trade-offs**: Increases frontend complexity (state syncing between chat and panel).
- **When to use**: Any AI application where the output is a functional document or application rather than just a textual answer.

**The Smart Router (Strategy Pattern)**
- **What**: An abstraction layer that selects the best LLM provider at runtime.
- **Why used here**: Allows users to use local models (Ollama) for privacy/speed and cloud models (Claude 3.5) for complex reasoning.
- **Trade-offs**: Requires maintaining multiple API integrations.
- **When to use**: When building "LLM-agnostic" tools where cost and latency vary across providers.

**NAPI-RS (Native Addons)**
- **What**: Using Rust to write Node.js modules for CPU-intensive tasks.
- **Why used here**: Parsing 50MB PDFs or complex HTML DOMs in TypeScript/JavaScript would block the event loop and be prohibitively slow.
- **Trade-offs**: Complex build pipeline (requires Rust toolchain).
- **When to use**: When JavaScript's performance bottleneck is the primary constraint.
